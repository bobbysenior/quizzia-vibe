# PR #2 — Plan : Couche API et logique métier

## Objectif

Mettre en place la couche métier pour l'authentification (sign up, sign in, sign out) et les quizz (créer, lister, voir un quizz). Pas de pages UI — uniquement la logique.

---

## Architecture

```
src/
├── lib/
│   ├── supabase/
│   │   ├── client.ts          # ✅ déjà créé — client navigateur
│   │   ├── server.ts          # ✅ déjà créé — client serveur
│   │   └── middleware.ts      # 🆕 client pour le middleware
│   ├── services/
│   │   └── quizzes.service.ts # 🆕 CRUD quizz (list, getById, create)
│   ├── auth/
│   │   └── actions.ts         # 🆕 Server Actions : signUp, signIn, signOut
│   └── types.ts               # 🆕 types partagés
├── app/
│   └── auth/
│       └── callback/
│           └── route.ts       # 🆕 callback OAuth/email
└── middleware.ts              # 🆕 rafraîchissement de session
```

### Décisions

- **Server Actions** pour les mutations auth : exécution côté serveur, sécurisé, pas de route REST à créer.
- **Service layer** (`services/`) : fonctions pures wrappant Supabase, réutilisables par Server Components et Server Actions.
- **Pas de route handlers REST** : Supabase PostgREST est déjà l'API. Les lectures passent par le service layer appelé depuis des Server Components. Les écritures par des Server Actions.
- **Middleware** : rafraîchit le cookie de session à chaque requête (obligatoire avec `@supabase/ssr`).

## Fichiers à créer (6)

### 1. `src/lib/types.ts` — Types partagés

```typescript
// Types métier correspondant au schéma DB
type QuizStatus = 'draft' | 'published' | 'archived';

interface Quiz {
  id: string; creator_id: string; title: string; theme: string;
  question_count: number; status: QuizStatus;
  created_at: string; updated_at: string;
}

interface QuizWithCount extends Quiz {
  actual_question_count: number;
}
```

### 2. `src/lib/supabase/middleware.ts` — Client pour middleware

Variante du client serveur adaptée au middleware : `createServerClient` avec gestion des cookies via `NextResponse` (pas `NextRequest`).

### 3. `src/middleware.ts` — Session refresh

- Appelé sur toutes les routes sauf `/_next/`, `/favicon.ico`, `/api/`
- Rafraîchit automatiquement `sb-*-auth-token`
- Pattern standard [Supabase Next.js docs](https://supabase.com/docs/guides/auth/server-side/nextjs)

### 4. `src/lib/services/quizzes.service.ts` — Service Quizz

| Fonction | Requête | RLS |
|---|---|---|
| `listQuizzes()` | `SELECT * FROM quizzes WHERE status = 'published'` | Public |
| `getQuizById(id)` | `SELECT quiz + COUNT(questions)` | Public + Creator |
| `createQuiz(data, userId)` | `INSERT INTO quizzes` | Authentifié |

- Fonctions pures, pas de JSX
- Validation Zod sur `createQuiz`

### 5. `src/lib/auth/actions.ts` — Server Actions Auth

```typescript
'use server';
// signUp(formData)  → supabase.auth.signUp() + redirection
// signIn(formData)  → supabase.auth.signInWithPassword() + redirection
// signOut()         → supabase.auth.signOut() + redirection
```

- Validation Zod des entrées (email, password)
- `signUp` avec `emailRedirectTo: '/auth/callback'`
- Retourne `{ error?: string }` pour affichage côté client

### 6. `src/app/auth/callback/route.ts` — Callback OAuth/Email

```typescript

```

## Flux d'exécution

```
Inscription :  signUp(email, password) → Auth → email → callback → session → /

Connexion :    signIn(email, password) → Auth → cookie → /

Créer quizz :  createQuiz(title, theme, count) → service → INSERT → draft

Voir quizz :   getQuizById(id) → service → SELECT + COUNT questions

Lister quizz : listQuizzes() → service → SELECT WHERE published
```
