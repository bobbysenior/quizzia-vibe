### 3. Rapport Backend (`Rapport Backend.md`)

- **Diagramme entité-relation complet** (Mermaid ER) : `auth.users` ↔ `quizzes` ↔ `questions` ↔ `choices` ↔ `quiz_attempts` ↔ `user_answers`.
- **Description exhaustive des 5 tables** :
  - `quizzes` : colonnes, types, `CHECK`, `ON DELETE`, trigger `set_updated_at`, index partiel sur `status = 'published'`.
  - `questions` : clé étrangère `quiz_id`, `order_index`, `ON DELETE CASCADE`.
  - `choices` : `is_correct`, `order_index`, trigger `check_single_correct_choice()` (garantie d’une seule bonne réponse par question).
  - `quiz_attempts` : support du jeu anonyme (`user_id` nullable), `ON DELETE CASCADE`, index sur `user_id` et `quiz_id`.
  - `user_answers` : dénormalisation de `is_correct`, contrainte UNIQUE `(attempt_id, question_id)`.
- **Triggers** : `set_updated_at()` et `check_single_correct_choice()` — code SQL et logique.
- **Row Level Security (RLS)** : toutes les policies détaillées table par table (`quizzes`, `questions`, `choices`, `quiz_attempts`, `user_answers`) avec règles et explications.
- **Historique des migrations** (5 migrations) :
  1. `20260516045551_initial_migration.sql` — création du schéma, index, triggers, RLS.
  2. `anonymous_play.sql` — support du jeu anonyme (`user_id` nullable).
  3. `fix_quiz_delete_cascade.sql` — ajout des `ON DELETE CASCADE` manquants.
  4. `quiz_select_attempted.sql` — policy `quizzes_select_attempted` pour liens historiques.
  5. `attempts_cascade_on_quiz_delete.sql` — confirmation des cascades.
- **Points de vigilance** : dénormalisation de `is_correct`, absence de soft delete, intégrité référentielle stricte.
