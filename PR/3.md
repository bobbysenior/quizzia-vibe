# PR #3 — Couche API et logique métier

## Résumé

Implémentation de la couche métier : authentification (sign up, sign in, sign out), session refresh, CRUD quizz. Les pages UI ne sont pas encore créées.

---

## Fichiers créés

| Fichier | Rôle |
|---|---|
| `src/lib/types.ts` | Types partagés : `Quiz`, `QuizWithCount`, `QuizStatus` |
| `src/lib/supabase/middleware.ts` | Client Supabase pour le proxy (gère les cookies via `NextResponse`) |
| `src/proxy.ts` | Session refresh automatique à chaque requête |
| `src/lib/services/quizzes.service.ts` | `listQuizzes()`, `getQuizById()`, `createQuiz()` |
| `src/lib/auth/actions.ts` | Server Actions : `signUp`, `signIn`, `signOut` (validation Zod) |
| `src/app/auth/callback/route.ts` | Échange code OAuth/email → session |
| `docs/api-guide.md` | Documentation pour les devs frontend |

---

## API exposée

### Auth (`@/lib/auth/actions`)

```
signUp(_prevState, formData)   → validation email/mdp → supabase.auth.signUp → redirect /login
signIn(_prevState, formData)   → validation email/mdp → supabase.auth.signInWithPassword → redirect /
signOut()                      → supabase.auth.signOut → redirect /
```

### Quizz (`@/lib/services/quizzes.service`)

```
listQuizzes()                  → SELECT published quizzes
getQuizById(id)                → SELECT quiz + COUNT questions
createQuiz({ title, theme, question_count, status? })  → INSERT (auth requise)
```

### Proxy

```
src/proxy.ts                   → Rafraîchit le cookie de session à chaque requête
```

---

## Conventions

- **Server Actions** : imports `'use server'`, utilisables avec `useActionState` côté client
- **Service layer** : fonctions async pures, créent leur propre client serveur Supabase
- **Auth** : Zod pour la validation, redirections intégrées dans les actions
- **Proxy** : Next.js 16 utilise `proxy.ts` (naming export `proxy`) au lieu de `middleware.ts`
